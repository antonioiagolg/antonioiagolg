# Simple WorkFlow for Agile
- A story should deliver something. This "something" should be testable somehow. It should be testable because the stakeholders expects a result which is valuable for them. This result should be the expected return of the story, so we need to prepare tests which garantee that this expected result will be reachable. Even we don't know where to start, or how we are going to achieve it, or if it is feasible, create a story that has the result if it is possible or not to achieve it.

- If we don't know if it is achievable or not, create a story that has the expected result a document with pros and cons about the desirable feature.

- Every story, given an expected result, should have the test scenarios. We don't need to add all possible tests, like unit tests, but scenarios that are readable by stakeholders. Most of the tools used for agile has the primary goal to inform our stakeholders about the work we are going to do, and how we are going to acomplish it. So, just write high level scenarios, which later will be translated by integration tests, or contract tests, or maybe unit tests. This technical complexity will be abstracted by oure high level scenarios.

- The story should solve a problem. Write the problem in the story description is a good starting point for refinement.
